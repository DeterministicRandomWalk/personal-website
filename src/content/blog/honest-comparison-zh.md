> 系列：我们构建了一条生产流水线。智能体为何做不到。

[上一篇：第3B篇——智能体会毁掉的90%。](/blog/agents-destroy-orchestration-zh/)  
[Read the independently edited English edition.](/blog/honest-comparison-en/)

同一个任务，确定性流水线的成本可以稳定在一个很低的基线；让通用智能体从头完成，成本往往会高出一个到两个数量级——如果它能顺利跑完的话。

前三篇讲了系统是怎么长出来的：分类为什么需要七层，LLM为什么只适合语义判断，以及链接提取、查询参数、停止规则、PDF处理和合规为什么必须写成确定性代码。

这一篇只做一件事：把两条路线摆在一起，逐项比较。

先说明口径：下面的数字是为了说明数量级的工程模型，不是供应商报价，也不是通用基准。模型假设智能体已经采用批处理、大上下文和较强模型，而不是最笨的“每个URL调用一次”。即使在这个相对有利的假设下，差距仍然很大。

## 大脑的成本

前文把分类系统比作大脑。关键词打分、域名前缀树、分数继承、PDF文件名评分和迭代缓存，会在调用LLM之前处理绝大多数明确情况。

| 成本项 | 确定性流水线 | 通用智能体 |
| :-- | :-- | :-- |
| 送进LLM的内容 | 经过多层预过滤后的含糊项 | 大量原始URL及页面上下文 |
| 首轮成本 | 小而有界 | 随网站规模线性增长 |
| 后续迭代 | 复用分类和路径缓存 | 容易重新支付上下文成本 |
| 模型职责 | 只判断真正含糊的语义 | 同时承担筛选、探索与控制 |
| 相对成本 | 1×基线 | 数十倍起，且上限难以预测 |

为什么差这么多？因为流水线在LLM开口之前已经处理了能够确定处理的部分。模型看到的不是整个网站，而是最后那一小块规则够不到的灰色地带。

通用智能体如果没有这些预过滤层，就只能把大量候选内容交给模型判断。网站越大，输入越多，成本越高。流水线的LLM成本则更接近“含糊程度”的函数，而不是“网站大小”的函数。

## 身体的成本

分类只是大脑。其余编排才是身体：链接提取、去重、查询参数规范化、停止规则、PDF调度、浏览器回退、状态管理和合规检查。

| 环节 | 确定性流水线 | 通用智能体 |
| :-- | :-- | :-- |
| 链接提取与去重 | 固定解析与规范化代码 | 需要工具调用和运行时判断 |
| 查询参数 | 规则引擎阻断重复与循环 | 漏做会爆炸，补做又增加推理 |
| 停止规则 | 有界预算和明确优先级 | 容易在“还有内容”中继续运行 |
| PDF处理 | 专用下载、验证和重试逻辑 | 每次失败都需要决定下一步 |
| PDF推迟与排序 | 统一调度，页面优先 | 看到链接就行动，顺序不稳定 |
| 浏览器回退 | 测一次、记住、复用 | 容易逐页重新试错 |
| 合规检查 | 请求前确定性过滤 | 每次运行重新解释约束 |

这些步骤在流水线中几乎不消耗模型token。它们当然有计算和浏览器成本，但行为稳定、可预算，而且随着规模扩大，边际成本可预测。

智能体路线的问题不是每一次调用都很贵，而是每一块“身体动作”都可能重新进入推理循环。失败会带来更多观察、更多上下文、更多工具调用和更多重试。费用不是简单相加，而是沿着失败路径扩散。

如果把流水线的端到端成本记作 `1×`，一个足够认真、但缺少专用工程层的智能体实现落在 `50×–100×` 并不奇怪。这个范围不是精确测量，而是由分类输入、浏览器交互、失败恢复和重复上下文共同构成的量级模型。

## 真正昂贵的是不完整

账单只是最容易看见的成本。

确定性流水线不需要“希望”模型注意到一个藏在通用内容路径下的报告。它可以根据链接来源、父页面分数和路径结构继承上下文。信息不是靠模型临场想起，而是沿着系统明确传播。

通用智能体面对一个扁平的候选集合，必须每次重新理解哪些内容重要。它通常能返回一些有用结果，但“找到一些”与“系统性地覆盖所有可发现内容”是两种不同的产品。

对于进入投资研究流程的数据，缺失往往比显性错误更危险。错误答案还能被质疑；没有被发现的文件不会举手告诉你它缺席了。

## 非确定性会复利

对一个短任务，智能体可能表现得很好。长流程的问题在于：可靠性是乘法，不是加法。

假设每个决策点都有95%的正确率，这已经是相当慷慨的假设。一个任务如果包含100个相互依赖的决策点：

```text
完整无误运行的概率 = 0.95^100 ≈ 0.6%
```

这不是说每次运行都会彻底失败。更常见的情况是：大多数步骤都对，少数步骤悄悄偏离。恰恰因为输出看起来大体合理，这类失败更难发现。

| 维度 | 确定性流水线 | 通用智能体 |
| :-- | :-- | :-- |
| 一致性 | 相同输入产生相同路径 | 运行间可能漂移 |
| 可复现性 | 可以重放状态和日志 | 无法保证相同决策 |
| 调试 | 定位规则、状态和失败点 | 解释一次生成式轨迹 |
| 完整性 | 可定义、测量和回归测试 | 容易停留在“看起来足够” |
| 合规性 | 每次请求执行相同检查 | 依赖模型记住并正确执行 |

查询参数循环、PDF静默失败和新闻无限探索并不是偶然的小错误。它们是缺少状态、预算和确定性边界后的结构性风险。

## 不要拿一次性任务冒充生产线

“找到某家公司的最新可持续发展报告。”

这是智能体擅长的任务：目标清楚、范围小、结果容易人工核实。

“系统性地覆盖一个大型公司集合的可发现披露，要求完整、可审计、可复现。”

这不是一个问题，而是一条生产线。

两者的差别不是模型够不够聪明，而是任务是否需要把经验固化成每次都执行的行为。一次性任务可以接受临场判断；生产线需要记忆、边界、状态和证据。

## 正确的组合

答案不是“智能体或者流水线”，而是智能体作为界面，流水线作为引擎。

```text
用户：为这家公司运行披露采集。
智能体：理解意图，补全必要参数，调用受控工具。
流水线：执行分类、采集、重试、状态和合规逻辑。
智能体：总结结果，突出失败项和需要人工判断的部分。
```

智能体负责理解意图、组织输入、解释输出。流水线负责那套不应该每天重新发明的确定性行为。双方都在自己擅长的位置上。

## “但智能体可以自己写代码”

可以，而且这正是它最有价值的地方之一。

AI编码工具能显著加快确定性流水线的实现、测试和重构。但“写代码很快”与“自动知道该写什么”不是一回事。

那些真正昂贵的发现通常来自真实运行：某类网站只在脚本渲染后出现链接，某些非标准标签藏着下载入口，查询参数形成循环，PDF看似打开却没有保存。只有问题浮出水面之后，工程师才能把边界描述清楚，让AI帮助实现修复。

代码生成能力越来越便宜。识别隐含需求、决定系统边界、判断什么必须确定执行，仍然是工程判断。

## 适用边界

智能体适合：

- **一次性任务**：构建专用流水线的成本高于几次模型调用。
- **探索阶段**：先了解网站和数据的大致形状。
- **代码生成**：帮助编写和维护确定性系统。
- **低风险、高多样性任务**：草稿、总结、初步审查等允许人工复核的工作。

确定性流水线更适合：

- **高吞吐量处理**：大量页面和文件需要一致处理。
- **高可靠性要求**：小概率错误会在长流程中复利。
- **有状态多步骤工作流**：任务需要跨阶段保存事实和决策。
- **零容忍约束**：合规、权限和预算边界必须每次执行。
- **高频微决策**：批次、超时、重试和限速不值得调用模型。

这不是模型能力的永久边界，但它是今天构建可靠系统时必须尊重的工程边界。
